Core Innovation

CPU Branch Prediction
for AI Agents

Modern CPUs don't stall on conditional branches — they speculate. AgentForge brings this same principle to multi-agent pipelines, turning sequential bottlenecks into parallelized pre-execution with bounded recovery on mispredict.

CPU Architecture
AgentForge Equivalent
Branch instruction — conditional branch whose outcome determines which path the CPU executes
Predecessor task whose output determines the input to the next agent in the DAG
Branch predictor — uses branch history table to guess the likely outcome before it is known
PredictionModel — frequency-based statistical predictor with historical learning and confidence scoring
Reorder buffer — holds speculative results in-flight until the branch resolves
BufferedMCPClient — reorder buffer pattern; holds tool calls in a side-effect buffer until validation
Pipeline flush on mispredict — all speculative work is discarded; CPU rewinds to branch point
RollbackCoordinator — discards buffered side effects and restores the pre-speculation checkpoint
Architectural commit — speculative results promoted to permanent architectural state
Buffered tool calls flushed from BufferedMCPClient to real MCP execution on successful validation

Implementation Detail

Component Deep Dive

Click any component to expand its design decisions, key mechanisms, and implementation highlights.

🗺
WorkflowEngine
DAG · Orchestrator
Kahn's algorithm for topological sort ensures correct task ordering. Parallel fan-out and fan-in use Java 21 Virtual Threads for lightweight concurrency without blocking OS threads.
  • Diamond DAG patterns fully supported — multiple predecessors converge correctly
  • Failure propagation cancels all downstream tasks on predecessor exception
  • StructuredTaskScope.ShutdownOnFailure ensures clean cancellation semantics
  • Fan-out spawns N speculative branches simultaneously; fan-in waits via CompletableFuture.allOf
⚡
SpeculationEngine
Speculation · Core
Orchestrates the full speculative execution lifecycle. Receives DAG tasks from WorkflowEngine and decides whether to speculate based on confidence threshold and depth limit.
  • Cascading confidence decay: confidence × 0.8^depth
  • Depth limit of 3 prevents unbounded speculative parallelism
  • Tracks per-task hit/miss counts; exposes hit rate stats for observability
  • On validation miss, delegates rollback to RollbackCoordinator
🧠
PredictionModel
ML · Prediction
Frequency-based statistical predictor. Maintains a historical table of task outputs per workflow context. No LLM call required for prediction — pure statistical inference.
  • Records outcome frequencies per (taskType, inputHash) key
  • Returns the most-frequent past output as predicted value
  • Confidence score = frequency / total_observations
  • Improves over time as more workflow runs are recorded
📦
BufferedMCPClient
Reorder Buffer · Side Effects
Implements the CPU reorder buffer pattern for AI agent side effects. All MCP tool calls during speculative execution are intercepted and held in memory — never committed to real state.
  • Wraps MCPClient; transparent to agent code via same interface
  • On commit: replays buffered calls to real MCPClient in order
  • On rollback: discards buffer — zero real-world side effects
  • Returns synthetic responses to agent during speculation (based on predicted state)
🔌
MCPClient
Protocol · Tool Integration
Full implementation of the Model Context Protocol over JSON-RPC 2.0. Enables agents to discover and invoke tools from any MCP-compliant server.
  • Tool discovery via tools/list — dynamic capability enumeration
  • Multi-server routing: agents select server by tool namespace prefix
  • Tool invocation via tools/call with structured parameter binding
  • Built on java.net.http.HttpClient with async non-blocking I/O
🤝
A2AClient
Protocol · Agent-to-Agent
Implements Google's Agent-to-Agent protocol for peer agent communication. Agents self-describe capabilities via Agent Cards; delegation is capability-driven.
  • Agent Card fetching — discovers skills, inputModes, outputModes
  • Capability matching: delegates tasks to best-fit peer by skill overlap
  • Full task lifecycle: submitted → working → completed/failed
  • Supports SSE streaming responses for long-running delegated tasks
💬
LLMClient
HTTP · LLM Integration
Zero-dependency LLM client built on java.net.http.HttpClient. Supports OpenAI-compatible APIs with streaming and automatic retry.
  • SSE streaming via BodyHandlers.ofLines() — no blocking reads
  • Exponential backoff retry with jitter on 429/503 responses
  • Jackson-based request/response serialization
  • Provider-agnostic: plug in OpenAI, Anthropic, or local Ollama endpoints
🚦
HITLGateManager
Human-in-the-Loop · Gates
Suspends workflow execution at configured checkpoints to await human approval. Gates are non-blocking — the workflow thread parks via CompletableFuture.
  • Gate configured per-task in workflow DAG definition
  • Pending approvals surfaced via REST API: GET /gates/pending
  • Approve/reject via POST /gates/{id}/approve or /reject
  • On reject, triggers RollbackCoordinator for the gated task subtree
📡
Kafka EventBus
Events · Exactly-Once
Exactly-once transactional event publishing for all workflow state transitions. Critical for audit trails and downstream consumers that cannot tolerate duplicates.
  • initTransactions() called once at producer startup
  • Each event batch wrapped in beginTransaction() / commitTransaction()
  • On exception, abortTransaction() prevents partial publishes
  • Event schema: workflowId, taskId, status, timestamp, payload
💾
CheckpointStore
State · Redis-style
Redis-backed persistence for agent state checkpoints. Used by RollbackCoordinator to restore pre-speculation state on validation failure.
  • Optimistic locking via CAS (compare-and-swap) prevents concurrent write conflicts
  • TTL expiry ensures stale checkpoints don't accumulate indefinitely
  • Checkpoint taken before speculation begins; keyed by (workflowId, taskId, speculationId)
  • Supports nested checkpoints for cascading speculation at depth 2 and 3

Maven Module Architecture

Clean separation of concerns across 5 modules with well-defined dependency boundaries.

common
Shared domain models, interfaces, and utility classes. Zero dependencies on other modules — the foundation everything else builds on.
AgentTask WorkflowResult Interfaces
orchestrator
Core orchestration logic: WorkflowEngine, SpeculationEngine, PredictionModel, BufferedMCPClient, RollbackCoordinator, CheckpointManager, HITL gates.
WorkflowEngine SpeculationEngine Virtual Threads
agent-pool
AgentRuntime implementations: SummarizerAgent, ClassifierAgent, MCPClient, A2AClient, LLMClient. Each agent is a stateless request handler.
AgentRuntime MCPClient A2AClient
gateway
REST API layer. Accepts workflow DAG submissions, exposes HITL gate endpoints, streams task status via SSE. Built with Java's built-in HTTP server.
REST API SSE Streaming HITL Endpoints
event-bus
Kafka producer/consumer wrappers with exactly-once transactional semantics. Publishes all workflow lifecycle events for audit and downstream consumers.
Kafka Exactly-Once Transactions
state-store
Redis-backed CheckpointStore with optimistic locking (CAS) and TTL expiry. Provides the durable state layer for speculation rollback and workflow recovery.
Redis CAS TTL Expiry
